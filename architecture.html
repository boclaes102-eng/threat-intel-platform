security model

Server-side proxy routes

src/app/api/*/route.ts

Every external API call goes through a Next.js server route. The browser never sees API keys or upstream URLs.

Example: /api/ip?ip=1.2.3.4 runs server-side, injects the AbuseIPDB key from env, returns enriched data.

why this shape Client bundles are inspectable. Any API key shipped to the browser is public. Keys live server-side, period.
+
edge layer

Sliding-window rate limiter

src/middleware.ts

Runs at Vercel's Edge before any function is invoked. Tracks timestamps per IP per route. Default: 60 req/min. Paid APIs: 20 req/min.

Rejected requests return 429 with Retry-After — never reach the serverless function. Zero compute cost.

why this shape Without rate limiting, one aggressive user can burn through free-tier quotas on paid APIs in minutes. In-memory Map scales per-instance; for multi-instance scale, swap for Redis.
+
type system

Hand-written TypeScript types

src/lib/types.ts · 278 lines

One interface per API response shape. No code generation, no auto-schema. Written by hand so every optional field is a deliberate decision.

When an external API changes its shape, TypeScript breaks the consumer — a build error, not a production bug.

why this shape Strict mode + discriminated unions force exhaustive handling. The compiler catches missing cases before a user does.

+
process topology

Two processes, shared data

src/index.ts + src/worker.ts

Split in two: a Fastify API answering HTTP, and a worker process running background jobs. They're separate Docker containers sharing one Postgres and one Redis.

The API must stay fast; the worker does slow, expensive things.

why this shape If heavy background work lived inside the API, a slow CVE sync would block user requests. Separation lets each process scale independently.
+
data model

10 tables with lifecycle states

src/db/schema/*

Users → assets → vulnerabilities (join table with status) → alerts. Plus ioc_records (TTL cache), feed_syncs (job history), recon_sessions (dashboard output).

Critical design choice: remediation status lives on the join table, not the vulnerability. The same CVE can be remediated on asset A and open on asset B.

  • All foreign keys have explicit ON DELETE actions (CASCADE or SET NULL)
  • Lifecycle states are Postgres enums — DB rejects invalid values
  • Passwords: bcrypt. API keys & refresh tokens: SHA-256 hashed
why this shape Status is a relation between a finding and an asset, not a property of the finding. This is the canonical model for any "continuous monitoring" SaaS.
+
enrichment pipeline

Cache → DB → parallel fan-out

src/services/ioc-enrichment.ts

When an indicator (IP, domain, hash) needs classification:

  • Check Redis cache (instant hit → return)
  • On miss: Promise.allSettled() to AbuseIPDB + VirusTotal + OTX in parallel
  • Combine verdicts, compute single score, write back to Redis with TTL

Cache hits and misses are Prometheus metrics — I can see the hit rate in Grafana.

why this shape allSettled over all: one slow source can't take down the others. Three sources cover different data — AbuseIPDB = community reports, VT = AV engines, OTX = threat pulses.
+
background workers

3 BullMQ queues on a schedule

src/workers/*

Three workers, three distinct jobs:

  • cve-feed — every 6h, pages through NIST NVD (2000+ pages), exponential backoff + jitter on rate limits
  • ioc-scan — every 1h, re-enriches active assets, creates alerts on malicious verdicts
  • asset-scan — on-demand when new asset registered, correlates against known CVEs

Each run is tracked in feed_syncs: duration, records processed, errors.

why this shape BullMQ over cron: persistence across restarts, automatic retries, admin UI. CVE sync mid-way restart = no data loss.
+
api surface

7 route groups · cursor pagination

src/api/routes/*

auth · assets · alerts · vulnerabilities · ioc · recon-sessions · health

All list endpoints use cursor pagination — no offset drift when data changes during pagination. Zod validates every request body and query param.

Request IDs traced through logs: single dashboard request → Next.js proxy log → Fastify log → worker log → email log, all linked.

why this shape Offset pagination drifts on live data (same row appears twice, rows skipped). Cursor pagination is stable.
+
observability

Prometheus + Grafana

src/lib/metrics.ts + grafana/

8-panel Grafana dashboard, pre-provisioned via Docker Compose:

+
execution model

In-process threading, not subprocess

launcher/utils/runner.py

Tools are loaded via importlib.util.spec_from_file_location() into the same Python interpreter. No child processes, no pipe juggling.

Each run lives in a daemon thread with try/except. Stop mechanism: set a cooperative flag, then inject KeyboardInterrupt via ctypes.PyThreadState_SetAsyncExc as fallback.

why this shape Subprocess is slower, harder to stop cleanly, and requires path juggling. In-process = faster + simpler. Daemon threads contain the blast radius of any crash.
+
output capture

Thread-aware stdout interceptor

launcher/utils/writer.py

Tools print() freely — no source changes. A custom writer wraps sys.stdout using threading.local(). Each worker thread carries its own GUI callback; other threads fall through to the original stream.

why this shape Alternative was to patch every print() call in every tool. This is zero-touch — tools stay testable from the CLI, GUI gets its output too.
+
backend bridge

Recon Workspace (online + offline)

launcher/pages/recon_page.py

Fetches saved recon sessions from the Railway backend. One click sets an active target → clipboard → paste into attack tool.

Config lives in ~/.cybersuite/config.json — never in source. If backend unreachable, the Offline tab is always available — operator types targets manually, everything still works.

why this shape Desktop tools run in the field where internet is unreliable. Graceful degradation is not optional. Online mode is the bonus, offline is the baseline.

The "why this stack" answers

Every technology choice is a trade-off. When asked "why X", the honest answer is usually "because the alternative had a specific problem I didn't want." Not hype-driven, not lock-in — deliberate.

Language

TypeScript over PHP / .NET

Same language front-to-back means shared types between frontend and backend. Native async I/O fits the fan-out pattern. Same stack modern Belgian SaaS runs on.

Framework

Next.js over plain React

Server-side proxy for credential hiding is native — one file, not a second deployment. Edge middleware runs before function invocation. Push-to-deploy on Vercel.

Backend

Fastify over Express

JSON Schema validation built-in. First-class TypeScript types. ~2x faster. Express's types are community-maintained; Fastify's are part of the framework.

Database

Postgres over Mongo

The data is relational: assets → CVEs → alerts. Foreign keys with explicit delete cascades are the feature, not tax. Postgres enums reject invalid values at the DB layer.

ORM

Drizzle over Prisma

Query API reads like SQL, so complex queries don't fight the ORM. Migrations are plain .sql files — readable, editable. Smaller bundle, faster cold starts.

Queue

BullMQ over node-cron

Cron has no persistence, no retries, no observability. BullMQ is Redis-backed — jobs survive restarts, retry with backoff, have an admin UI. Right tool for production.

Cache

Redis necessary, and useful

BullMQ requires Redis — not optional. Once it's there, native TTL expiry for the IOC cache is free. Two databases is overhead, but worth it.

Hosting

Railway + Vercel push-to-deploy

Railway gives two Docker targets (API + worker) sharing Postgres and Redis, one config. Vercel gives Next.js Edge features. Both deploy on git push.

Same architecture as continuous monitoring SaaS products like Webclew — different domain.

Both systems do the same structural work: register a target, scan continuously in the background, classify what's found, alert on policy violations, track remediation through a lifecycle. The difference is the classification layer — not the plumbing around it.

Identical structure
  • Register target → async workers scan → findings stored with severity + lifecycle
  • External data sources feed the classifier in parallel with cache fallback
  • Alert generation with severity gating (not every finding becomes a notification)
  • Remediation lifecycle on the join table — same finding can be open on one target, resolved on another
  • Cursor pagination, enum-enforced states, explicit foreign key behavior
  • Server-side proxy hides backend credentials from the browser
What's different
  • My classifier hits JSON APIs (NVD, VT, AbuseIPDB) — theirs runs headless browsers to observe real network traffic
  • My rules are security-focused (CVSS severity, threat feeds) — theirs are legally framed (GDPR, TCF, DPAs)
  • My alerts describe a match — theirs may become evidence in a regulator case, so the proof surface is heavier
  • My customer is a security operator — theirs is a Data Protection Officer
